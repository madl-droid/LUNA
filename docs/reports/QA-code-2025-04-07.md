## Bugs encontrados

### BUG-1: YouTube usa API key de Gemini en vez de OAuth (403)
**Severidad**: Alta — YouTube knowledge no funciona
**Archivo**: `src/extractors/youtube-adapter.ts`, `src/modules/knowledge/item-manager.ts`
**Síntoma**: Al entrenar un item tipo YouTube (playlist/video/channel), la YouTube Data API v3 devuelve 403.
**Causa raíz**: `item-manager.ts` pasa `this.config.KNOWLEDGE_GOOGLE_AI_API_KEY` como apiKey a `listPlaylistVideos()`, `getVideoMeta()`, `getChannelMeta()`. Esa key es una Google AI Studio key (para Gemini/embeddings), que NO tiene habilitado el servicio YouTube Data API v3. El sistema ya tiene OAuth conectado con scope `youtube.readonly` pero el youtube-adapter no lo usa.
**Log evidencia**:
```
[YT] Playlist API error — status: 403, playlistId: PLqMtO-RRUphCUUXIulnP5xPo7SZ8Cgisu
[YT-PLAYLIST] Videos found — videoCount: 0
Content loaded — chunks: 0
```
**Fix sugerido**:
1. En `youtube-adapter.ts`: agregar parámetro opcional `opts?: { accessToken?: string }` a `getVideoMeta`, `listPlaylistVideos`, `getChannelMeta`. Cuando se pasa, usar `Authorization: Bearer` en vez de `key=` en query string.
2. En `item-manager.ts`: obtener access token via `this.registry.getOptional('google:oauth-client')?.credentials?.access_token` y pasarlo como `{ accessToken }` a todas las llamadas YouTube.
3. Backward compatible: si no hay OAuth, sigue intentando con API key.

### BUG-2: Botón "Entrenar agente" (bulk) no carga contenido pendiente
**Severidad**: Media — confunde al usuario, items quedan sin entrenar
**Archivo**: `src/modules/knowledge/console-section.ts` (línea ~1140), `src/modules/knowledge/knowledge-manager.ts`
**Síntoma**: El botón "Entrenar agente" se muestra activo (naranja) cuando hay items con `embedding_status !== 'embedded'`, pero al hacer click solo llama `triggerBulkVectorization()` que re-embede chunks existentes. Items con `content_loaded=false` se ignoran.
**Causa raíz**: `kiBulkVectorize()` en el frontend hace `POST /vectorize` que llama `triggerBulkVectorization()` → `vectorizeWorker.enqueueBulk()`. Esto solo procesa chunks ya guardados en DB, no llama `loadContent()` para items pendientes.
**Fix sugerido**: En `triggerBulkVectorization()` o en el handler de `POST /vectorize`, primero buscar items con `content_loaded=false AND active=true` y llamar `loadContent()` para cada uno antes de re-vectorizar.

### BUG-3: UI no refleja estado real de embedding sin reload manual
**Severidad**: Media — UX confusa
**Archivo**: `src/modules/knowledge/console-section.ts`
**Síntoma**: Después de entrenar, el status dot muestra "Procesando" indefinidamente. El usuario debe recargar la página manualmente para ver "Entrenado". No hay feedback del progreso de embedding (que puede tardar varios minutos para items grandes).
**Fix sugerido**:
1. Agregar endpoint `GET /items/progress?id=` que consulte `SELECT COUNT(*) FILTER (WHERE embedding_status='embedded') as done, COUNT(*) as total FROM knowledge_chunks ...`.
2. Después de `loadContent`, hacer polling cada 3s al endpoint de progreso.
3. Mostrar inline "X/Y (Z%)" en la fila del item durante embedding.
4. Cuando `done === total`, actualizar status dot a "Entrenado" sin reload.

### BUG-4: Módulo scheduled-tasks crashea al activarse
**Severidad**: Baja — no impacta knowledge ni chat, pero el módulo no funciona
**Log evidencia**:
```
kernel:loader module=scheduled-tasks "Failed to activate module"
Error: Invalid explicit day of month definition
  at CronExpression._handleMaxDaysInMonth (cron-parser/lib/expression.js:391)
```
**Causa raíz**: Una expresión cron inválida en la configuración de scheduled-tasks. Posiblemente un job con día de mes fuera de rango.
**Fix sugerido**: Revisar las expresiones cron registradas en `scheduled-tasks` y validar/corregir la que tiene día inválido.
