# Plan 2: Pipeline de Documentos + Tools

## Objetivo
Implementar el flujo completo de creación de documentos desde plantillas: tools para el agente, creación (copy → fill → organize → share), re-edición in-place, gestión de carpetas, sharing, strict mode, HITL.

## Prerequisitos
- **Plan 1 completado**: módulo `templates` existe con DB, repository, service base, console UI, Drive extensions (`copyFile`, `shareFileAnyone`)

## Contexto
El agente necesita tools para:
1. Crear documentos desde plantillas (el flow principal)
2. Buscar documentos ya generados (evitar duplicados, re-compartir)
3. Re-editar documentos existentes (cambiar valores de keys)

Los tools se registran siguiendo el patrón existente: el módulo `templates` registra tools via `tools:registry` en su `init()`.

## Paso 1: Folder Manager

### 1.1 Crear `src/modules/templates/folder-manager.ts`

Responsabilidad: resolver folder patterns a Drive folder IDs, creando carpetas si no existen, SIN duplicados.

```typescript
export class FolderManager {
  constructor(
    private drive: DriveService,
    private rootFolderId: string,
  ) {}

  /**
   * Resuelve un folder pattern a un Drive folder ID.
   * Crea carpetas que no existan. Nunca duplica.
   *
   * @param pattern - ej: "Comparativos/{BRAND}/{COMPETITOR}"
   * @param values  - ej: { BRAND: "Nike", COMPETITOR: "Adidas" }
   * @returns Drive folder ID de la carpeta final
   */
  async resolveFolder(pattern: string, values: Record<string, string>): Promise<string>
}
```

**Algoritmo de `resolveFolder`:**
1. Reemplazar placeholders en el pattern: `"Comparativos/{BRAND}/{COMPETITOR}"` → `"Comparativos/Nike/Adidas"`
2. Split por `/` → `["Comparativos", "Nike", "Adidas"]`
3. Empezar desde `rootFolderId`
4. Para cada segmento:
   a. Buscar en Drive: `drive.listFiles({ folderId: currentParent, query: segmentName, mimeType: 'application/vnd.google-apps.folder' })`
   b. Filtrar resultados: buscar match EXACTO de nombre (listFiles usa `contains`, no exact match)
   c. Si encontrado → usar ese folder ID como nuevo parent
   d. Si no encontrado → `drive.createFolder(segmentName, currentParent)` → usar nuevo ID
5. Retornar el folder ID final

**Trampa importante**: `drive.listFiles` usa `name contains 'X'` que matchea parcialmente. Ejemplo: buscar "Nike" también matchea "Nike Pro". El filtrado exacto debe hacerse client-side comparando `file.name === segmentName`.

**Cache en memoria** (opcional pero recomendado):
- Map<string, string> donde key = path completo (ej: "Comparativos/Nike/Adidas"), value = folder ID
- Invalidar al recibir hook `console:config_applied` (por si cambia root folder)
- TTL: vida del proceso (las carpetas no se mueven)

## Paso 2: Document Creator

### 2.1 Extender `src/modules/templates/service.ts` con métodos de creación

```typescript
// Agregar a TemplatesService:

/**
 * Crea un documento desde una plantilla.
 * Flow: copy template → fill keys → move to folder → share → track in DB
 */
async createDocument(input: {
  templateId: string
  keyValues: Record<string, string>
  contactId?: string
  requesterSenderId?: string
  requesterChannel?: string
  docName: string
  tags?: Record<string, string>
}): Promise<DocGenerated>

/**
 * Re-edita un documento existente (in-place, mismo link).
 * Flow: replaceText para cada key cambiada → update DB → increment version
 */
async reeditDocument(input: {
  generatedDocId: string
  updatedKeyValues: Record<string, string>  // solo los keys que cambian
}): Promise<DocGenerated>

/**
 * Busca documentos generados existentes.
 * Usado para evitar duplicados (especialmente comparativos).
 */
async findExistingDocument(query: {
  docType?: DocType
  tags?: Record<string, string>
  contactId?: string
}): Promise<DocGenerated[]>
```

### 2.2 Lógica de `createDocument`

```
1. Obtener template desde DB
2. Validar que template está enabled
3. Validar que todos los keys del template tienen valor en keyValues
   - Si faltan keys → error descriptivo
4. Obtener Drive service: registry.get<DriveService>('google:drive')
5. Resolver carpeta destino:
   - folderManager.resolveFolder(template.folderPattern, keyValues)
   - Si folderPattern está vacío, usar rootFolderId directamente
6. Copiar template: drive.copyFile(template.driveFileId, docName, folderId)
7. Rellenar keys según mimeType:
   - document: Para cada key → docsService.replaceText(newFileId, `{${key}}`, value)
     O mejor: docsService.batchEdit(newFileId, keys.map(k => ({ type: 'replace', search: `{${k.key}}`, replace: keyValues[k.key] })))
   - presentation: slidesService.batchEdit(newFileId, keys.map(k => ({ type: 'replace_text', search: `{${k.key}}`, replace: keyValues[k.key] })))
   - spreadsheet: sheetsService.batchEdit(newFileId, keys.map(k => ({ type: 'find_replace', find: `{${k.key}}`, replacement: keyValues[k.key] })))
8. Compartir: drive.shareFileAnyone(newFileId, 'reader')
9. Obtener webViewLink: drive.getFile(newFileId) → file.webViewLink
10. Guardar en DB: repository.createGenerated(db, { templateId, driveFileId: newFileId, ... })
11. Retornar DocGenerated
```

**Importante**: Usar batch edit (un solo API call con todas las replacements) en vez de N llamadas individuales. Los 3 servicios soportan batch.

### 2.3 Lógica de `reeditDocument`

```
1. Obtener doc generado desde DB (con key_values actuales)
2. Obtener template desde DB (para saber el mimeType)
3. Calcular diff: ¿qué keys cambiaron?
   - changedKeys = Object.entries(updatedKeyValues).filter(([k, v]) => currentKeyValues[k] !== v)
4. Si no hay cambios → retornar doc sin modificar
5. **Conflict check**: verificar que ningún par de keys con valores distintos comparta el mismo old_value
   - oldValues = changedKeys.map(([k]) => currentKeyValues[k])
   - Si hay duplicados en oldValues → flag conflicto
6. Si NO hay conflicto (caso normal):
   - Para cada changed key: replaceText(docFileId, oldValue, newValue)
   - Usar batch edit si disponible
7. Si HAY conflicto (caso raro):
   - Fallback: regenerar documento completo
   - Obtener template → copiar fresh → aplicar TODOS los key values (merged: current + updated)
   - Exportar como Office format → files.update() sobre el file ID original como nueva revisión
   - NOTA: esto requiere agregar un método `updateFileContent(fileId, content, mimeType)` al Drive service
   - Si el fallback de versión no es viable técnicamente, crear doc nuevo y archivar el anterior
8. Merge key_values: { ...currentKeyValues, ...updatedKeyValues }
9. Actualizar DB: repository.updateGenerated(db, id, { keyValues: merged, version: current + 1 })
10. Retornar DocGenerated actualizado
```

**Para el fallback de conflicto**, agregar a `drive-service.ts`:
```typescript
async updateFileContent(
  fileId: string,
  content: Buffer,
  mimeType: string,
): Promise<void> {
  await this.drive.files.update({
    fileId,
    media: { mimeType, body: Readable.from([content]) },
  })
}
```

Esto permite subir nuevo contenido al mismo file ID. Para Google Workspace files, subir un .docx con el mimeType correcto y Google lo convierte automáticamente, creando una nueva revisión.

## Paso 4: Template Catalog Injection (Prompt)

### 4.1 Inyectar catálogo de plantillas en el prompt del agente

El agente necesita SABER qué plantillas existen para poder decidir cuándo usarlas.

**Mecanismo**: Proveer un servicio `templates:catalog` que retorna texto formateado para inyección en prompt.

```typescript
// En service.ts:
async getCatalogForPrompt(): Promise<string> {
  const templates = await repository.listTemplates(this.db, { enabled: true })
  if (templates.length === 0) return ''

  const lines = templates.map(t => {
    const keyList = t.keys.map(k => `${k.key}: ${k.description}`).join(', ')
    return `- ${t.name} (${t.docType}, ${t.mimeType}): keys=[${keyList}]`
  })

  return `## Plantillas de documentos disponibles\n${lines.join('\n')}\n\nUsa la herramienta create-from-template para crear documentos. SIEMPRE busca documentos existentes antes de crear uno nuevo (search-generated-documents).`
}
```

**Inyección en prompt**: Registrar hook `message:before_respond` o proveer el servicio para que el engine lo consulte.

La forma más limpia (siguiendo el patrón de HITL rules y knowledge injection):
1. Proveer servicio: `registry.provide('templates:catalog', { getCatalogText: () => service.getCatalogForPrompt() })`
2. El engine (context builder) consulta `registry.getOptional('templates:catalog')` y lo inyecta en el system prompt

**NOTA**: Esto requiere una modificación mínima en el engine's context builder (`src/engine/prompts/context-builder.ts` o similar). Buscar dónde se inyecta el HITL rules text y seguir el mismo patrón para templates catalog.

### 4.2 Inyección de strict mode en prompt

Si strict mode está activo, inyectar regla en el prompt:

REGLA: Solo puedes crear documentos usando las plantillas registradas. NO crees documentos sin plantilla.

## Paso 5: Sharing Logic

### 5.1 Integrar en `createDocument`

Después de crear el documento:
1. SIEMPRE: `drive.shareFileAnyone(newFileId, 'reader')` — hace el link público (view only)
2. El agente comparte el `webViewLink` en su respuesta al contacto

### 5.2 Restricciones comportamentales

Las restricciones de "solo compartir con quien pidió" son COMPORTAMENTALES (en el prompt/tool guidance), no técnicas:

**En detailedGuidance de `search-generated-documents`:**
```
REGLA DE COMPARTIR:
- Comparativos: puedes compartir el enlace con cualquier contacto que lo solicite.
- Cotizaciones y presentaciones: SOLO comparte el enlace con:
  1. El contacto que originalmente solicitó el documento
  2. Coworkers internos que lo soliciten
  NO compartas cotizaciones ni presentaciones con otros contactos externos.
```

**En detailedGuidance de `create-from-template`:**
```
ANTES de crear un documento:
1. Busca si ya existe uno similar con search-generated-documents
2. Si existe un comparativo del mismo producto/competencia, comparte el existente
3. Para cotizaciones/presentaciones, cada contacto puede tener la suya propia
```

// Obtener hitl:manager service
const hitlManager = registry.getOptional<HitlManager>('hitl:manager')
if (hitlManager) {
  await hitlManager.createTicket({
    contactId: context.contactId,
    channel: context.channelName,

return {
  success: false,
  message: 'Estoy consultando internamente para poder ayudarte con ese documento.',

}
