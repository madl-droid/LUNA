## HUECO-1: Humanos asignados pueden cerrar compromisos desde conversación reactiva

### Problema
Cuando un humano recibe un compromiso via HITL handoff, Luna le envía un mensaje tipo "Recordatorio de compromiso... responde 'completado'". El humano responde, pero el pipeline reactivo no sabe que este usuario tiene compromisos asignados — el context builder solo inyecta compromisos **del contacto** (`WHERE contact_id = $1`), no compromisos **asignados al sender**.

### Diseño
1. En intake (`boundaries/intake.ts`), si el sender NO es un lead (es coworker/admin/etc), cargar compromisos asignados a él via una nueva query.
2. Inyectarlos en `ctx.pendingCommitments` junto con los del contacto (marcados distinto).
3. El LLM ya conoce `update_commitment` — al ver "Compromisos asignados a ti" + mensaje "completado", usará la tool.

## HUECO-2: Validación de JSON en KNOWLEDGE_CONTACT_CATEGORY_MAP

### Problema
El campo `KNOWLEDGE_CONTACT_CATEGORY_MAP` es `z.string().default('')` sin validación de JSON. Si alguien pone JSON inválido en la consola, el sistema falla silenciosamente (fail-open). Mejor validar en save para feedback inmediato.

### Diseño
Validar en el configSchema con `.refine()`. Si el valor es vacío → ok. Si no es vacío → debe ser JSON válido con estructura `Record<string, string[]>`.

### Cambios

#### Archivo: `src/modules/knowledge/manifest.ts`

Cambiar la línea del configSchema (~línea 1166):

```typescript
// ANTES
KNOWLEDGE_CONTACT_CATEGORY_MAP: z.string().default(''),

// DESPUÉS
KNOWLEDGE_CONTACT_CATEGORY_MAP: z.string().default('').refine(
  (v) => {
    if (!v) return true
    try {
      const parsed = JSON.parse(v)
      if (typeof parsed !== 'object' || parsed === null || Array.isArray(parsed)) return false
      return Object.values(parsed).every(arr => Array.isArray(arr) && arr.every(id => typeof id === 'string'))
    } catch { return false }
  },
  { message: 'Must be valid JSON: {"contact_type": ["category-id", ...]} or empty' },
),
```

**Nota**: `refine()` se ejecuta DESPUÉS de `.default('')`, así que el default vacío pasa la validación. Solo se rechaza si alguien pone JSON malformado. El error se surfacea en la consola al guardar.

### Problema

## DEUDA-4: Context summary mejorado con `context_note` del LLM

### Problema
El context_summary son los últimos 6 mensajes truncados. Funciona en ~80% de casos, pero si la conversación es larga, los 6 últimos mensajes pueden no contener el contexto relevante. Un resumen del LLM sería mejor, pero agregar otra llamada LLM es costoso.

### Diseño
**Solución zero-cost**: El LLM ya está en el loop cuando llama a `create_commitment`. Agregar un parámetro opcional `context_note` que el LLM llena con un breve resumen de por qué se crea el compromiso. Si el LLM lo provee, se usa como context_summary primario. Los mensajes raw se guardan como fallback.

### Cambios

  },
