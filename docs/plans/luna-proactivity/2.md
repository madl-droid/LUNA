# Plan 2 — Commitments overhaul + HITL handoff

## Objetivo
Hacer que los compromisos sean auto-contenidos, se cumplan siempre, y que cuando un humano toma responsabilidad Luna le haga seguimiento.

## Contexto
- El commitment auto-detector (Via B) gasta 1 LLM call por cada respuesta del agente. Lo eliminamos.
- Los compromisos no guardan contexto de la conversación donde se crearon. Cuando se disparan, el LLM tiene solo 5 mensajes de historial y no sabe qué pasó.
- El campo `assigned_to` existe en la tabla `commitments` pero no se conecta con HITL.
- El reminder.ts ya notifica a salespersons, pero el commitment-check.ts no distingue entre compromisos del agente vs compromisos delegados a humanos.

## Dependencia
- **Requiere Plan 1 completado** (bugs que impiden arrancar el sistema proactivo).

**Agregar función helper:**

```typescript
/**
 * Reassign pending commitments from the agent to a human who took responsibility.
 * The commitment scanner will then notify the human instead of the contact.
 */
async function reassignCommitmentsToHuman(
  db: Pool,
  contactId: string,
  humanId: string,
  humanChannel: string,
): Promise<void> {
  try {
    const result = await db.query(
      `UPDATE commitments
       SET assigned_to = $1, metadata = metadata || $2, updated_at = now()
       WHERE contact_id = $3
         AND status IN ('pending', 'in_progress', 'waiting', 'overdue')
         AND assigned_to IS NULL`,
      [humanId, JSON.stringify({ assigned_channel: humanChannel, assigned_via: 'hitl_handoff' }), contactId],
    )
    if (result.rowCount && result.rowCount > 0) {
      logger.info({ contactId, humanId, count: result.rowCount }, 'Commitments reassigned to human via HITL handoff')
    }
  } catch (err) {
    logger.warn({ err, contactId, humanId }, 'Failed to reassign commitments to human')
  }
}
```

```typescript
/**
 * Notify the assigned human that their commitment is due/overdue.
 * Uses message:send hook to contact the human directly.
 */
async function notifyAssignedHuman(ctx: ProactiveJobContext, row: Record<string, unknown>): Promise<void> {
  const assignedTo = row.assigned_to as string
  const metadata = (row.metadata ?? {}) as Record<string, unknown>
  const channel = (metadata.assigned_channel as string) || 'whatsapp'
  const description = row.description as string
  const status = row.status as string
  const commitmentId = row.id as string
  const contactName = row.display_name as string | null
  const attemptCount = (row.attempt_count as number) ?? 0

  const maxAttempts = ctx.proactiveConfig.commitments.max_attempts

  // Build notification message
  const overdueTag = status === 'overdue' ? ' ⚠ VENCIDO' : ''
  const message = `📋 Recordatorio de compromiso${overdueTag}\n\nContacto: ${contactName ?? 'Sin nombre'}\nCompromiso: ${description}\nIntento: ${attemptCount + 1}/${maxAttempts}\n\nPor favor atiende este compromiso o responde "completado" cuando lo hayas resuelto.`

  try {
    await ctx.registry.runHook('message:send', {
      channel,
      to: assignedTo,
      content: { type: 'text', text: message },
      correlationId: ctx.traceId,
    }, ctx.traceId)

    // Update attempt count
    await ctx.db.query(
      `UPDATE commitments SET attempt_count = attempt_count + 1, last_attempt_at = now()
       WHERE id = $1`,
      [commitmentId],
    )

    logger.info({ commitmentId, assignedTo, attempt: attemptCount + 1 }, 'Notified assigned human about commitment')
  } catch (err) {
    logger.warn({ err, commitmentId, assignedTo }, 'Failed to notify assigned human')
  }

  // If max attempts reached, escalate (re-assign to agent or mark failed)
  if (attemptCount + 1 >= maxAttempts) {
    await ctx.db.query(
      `UPDATE commitments SET status = 'failed', action_taken = 'Human did not respond after max attempts'
       WHERE id = $1`,
      [commitmentId],
    )
    logger.warn({ commitmentId, assignedTo }, 'Human commitment failed after max attempts — consider escalation')

    // Optionally create HITL ticket to escalate
    const hitlManager = ctx.registry.getOptional<{
      createTicket(params: Record<string, unknown>): Promise<unknown>
    }>('hitl:manager')
    if (hitlManager) {
      try {
        await hitlManager.createTicket({
          requestType: 'escalation',
          requestSummary: `Compromiso no atendido por ${assignedTo}: "${description}"`,
          requesterContactId: row.contact_id as string,
          urgency: 'high',
          targetRole: 'admin',
          metadata: { commitmentId, originalAssignee: assignedTo },
        })
      } catch (err) {
        logger.warn({ err, commitmentId }, 'Failed to create escalation ticket for unattended commitment')
      }
    }
  }
}
```

## Cambio 6: update_commitment permite que el humano cierre compromisos

### Qué hacer
El tool `update_commitment` ya permite cambiar status a 'done'. Pero cuando un humano responde a la notificación con "completado" o "listo", Luna necesita interceptar y cerrar el compromiso.

### Enfoque
Esto ya funciona via el flujo normal: cuando el humano (que es un admin/coworker) escribe "completado", Luna ve los compromisos pendientes (via context-builder) y puede usar `update_commitment`. Solo necesitamos que el prompt proactivo sea claro.

### Archivo: `instance/prompts/system/proactive-agentic-system.md`

Agregar al final de las instrucciones de compromiso:

```markdown
## Compromisos asignados a humanos
- Si el contacto es un admin/coworker y tiene compromisos asignados, recuérdales amablemente.
- Si responden "completado", "listo", "ya lo hice", "done", o similar → usa update_commitment con status=done.
- Si responden "no puedo", "no me es posible" → usa update_commitment con status=cancelled y crea un nuevo compromiso para que el agente lo maneje.
```
