## Fix 1: Eliminar migración duplicada 048_ (BUG-1, CRÍTICO)

### Problema
Dos archivos con prefijo `048_`: `048_commitment-context-summary.sql` y `048_follow-up-intensity.sql`. El primero ya incluye el `ALTER TABLE agent_contacts ADD COLUMN IF NOT EXISTS follow_up_intensity`, haciendo al segundo 100% redundante. Tener dos migraciones con el mismo número viola el contrato del migrador.

### Acción
**Eliminar** el archivo `src/migrations/048_follow-up-intensity.sql` completamente.

```bash
rm src/migrations/048_follow-up-intensity.sql
```

El contenido ya está cubierto por las líneas 6-7 de `src/migrations/048_commitment-context-summary.sql`:
```sql
-- Follow-up intensity per contact (used by Plan 3, created here for single migration)
-- Values: 'aggressive', 'normal', 'gentle', 'minimal'
ALTER TABLE agent_contacts ADD COLUMN IF NOT EXISTS follow_up_intensity TEXT NOT NULL DEFAULT 'normal';
```

## Fix 3: Arreglar `ensureInstanceFiles()` — source ≠ target (BUG-3, LÓGICO)

### Problema
`src/kernel/bootstrap.ts` líneas 25-26 tiene `source` y `target` apuntando al mismo path (`instance/proactive.json`). Si un Docker volume mount sobreescribe `instance/`, tanto source como target desaparecen. La función no puede restaurar el archivo.

### Evaluación
`loadProactiveConfig()` en `src/engine/proactive/proactive-config.ts` ya tiene un `DEFAULT_CONFIG` hardcoded completo y maneja gracefully el caso donde `instance/proactive.json` no existe (líneas 35-38). El sistema funciona sin el archivo.

### Acción
**Eliminar** la entrada de `proactive.json` de `REQUIRED_FILES` ya que la función no puede cumplir su propósito (no hay template separado) y el loader ya tiene defaults robustos.

### Archivo: `src/kernel/bootstrap.ts`

```typescript
// ANTES (líneas 24-27)
const REQUIRED_FILES: Array<{ target: string; source: string }> = [
  { target: 'instance/proactive.json', source: 'instance/proactive.json' },
]

// DESPUÉS
const REQUIRED_FILES: Array<{ target: string; source: string }> = [
  // proactive.json: not needed here — loadProactiveConfig() has hardcoded defaults
  // and gracefully handles missing file. Add future entries that have a real
  // source template (outside instance/) here.
]
```

La función `ensureInstanceFiles()` se mantiene (puede servir para futuros archivos con templates reales), pero el array queda vacío por ahora.

## Fix 4: Mover check de `max_attempts` dentro del `try` en `notifyAssignedHuman` (BUG-4, FUNCIONAL)

### Problema
En `src/engine/proactive/jobs/commitment-check.ts`, la función `notifyAssignedHuman()` tiene el check de `attemptCount + 1 >= maxAttempts` **fuera** del `try/catch`. Si `message:send` falla, el `attempt_count` no se incrementa en DB (el UPDATE está dentro del try), pero el check de max_attempts usa el valor pre-computado. Si ya estaba en el último intento al momento del fallo, marca `failed` sin haber logrado notificar nunca.

### Archivo: `src/engine/proactive/jobs/commitment-check.ts`

**Localizar** la función `notifyAssignedHuman` (comienza ~línea 167). Mover el bloque de max_attempts dentro del `try`, después del UPDATE exitoso:

```typescript
// ANTES (líneas 182-230 aprox)
  try {
    await ctx.registry.runHook('message:send', {
      channel,
      to: assignedTo,
      content: { type: 'text', text: message },
      correlationId: ctx.traceId,
    }, ctx.traceId)

    await ctx.db.query(
      `UPDATE commitments SET attempt_count = attempt_count + 1, last_attempt_at = now()
       WHERE id = $1`,
      [commitmentId],
    )

    logger.info({ commitmentId, assignedTo, attempt: attemptCount + 1 }, 'Notified assigned human about commitment')
  } catch (err) {
    logger.warn({ err, commitmentId, assignedTo }, 'Failed to notify assigned human')
  }

  // If max attempts reached, escalate
  if (attemptCount + 1 >= maxAttempts) {
    await ctx.db.query(
      `UPDATE commitments SET status = 'failed', action_taken = 'Human did not respond after max attempts'
       WHERE id = $1`,
      [commitmentId],
    )
    logger.warn({ commitmentId, assignedTo }, 'Human commitment failed after max attempts')

    const hitlManager = ctx.registry.getOptional<{
      createTicket(params: Record<string, unknown>): Promise<unknown>
    }>('hitl:manager')
    if (hitlManager) {
      // ... hitlManager.createTicket(...) ...
    }
  }
}
```

```typescript
// DESPUÉS
  try {
    await ctx.registry.runHook('message:send', {
      channel,
      to: assignedTo,
      content: { type: 'text', text: message },
      correlationId: ctx.traceId,
    }, ctx.traceId)

    await ctx.db.query(
      `UPDATE commitments SET attempt_count = attempt_count + 1, last_attempt_at = now()
       WHERE id = $1`,
      [commitmentId],
    )

    logger.info({ commitmentId, assignedTo, attempt: attemptCount + 1 }, 'Notified assigned human about commitment')

    // Check max attempts AFTER successful notification + DB update
    if (attemptCount + 1 >= maxAttempts) {
      await ctx.db.query(
        `UPDATE commitments SET status = 'failed', action_taken = 'Human did not respond after max attempts'
         WHERE id = $1`,
        [commitmentId],
      )
      logger.warn({ commitmentId, assignedTo }, 'Human commitment failed after max attempts')

      const hitlManager = ctx.registry.getOptional<{
        createTicket(params: Record<string, unknown>): Promise<unknown>
      }>('hitl:manager')
      if (hitlManager) {
        // ... hitlManager.createTicket(...) — keep existing code unchanged ...
      }
    }
  } catch (err) {
    logger.warn({ err, commitmentId, assignedTo }, 'Failed to notify assigned human — will retry on next scan')
  }
}
```

**Clave del cambio**: El bloque `if (attemptCount + 1 >= maxAttempts) { ... }` se mueve DENTRO del `try`, después del UPDATE exitoso. El `catch` ahora abarca todo — si la notificación falla, ni se incrementa el counter ni se marca como failed. El siguiente scan lo intentará de nuevo.

## Fix 5: Incrementar `processed` para commitments de humanos (BUG-5, MENOR)

### Problema
En `src/engine/proactive/jobs/commitment-check.ts`, los commitments asignados a humanos se procesan via `notifyAssignedHuman()` y luego `continue`, sin incrementar `processed`. El log final reporta un count incorrecto.

### Archivo: `src/engine/proactive/jobs/commitment-check.ts`

**Localizar** el bloque que maneja commitments humanos (~líneas 52-55):

```typescript
// ANTES
      if (row.assigned_to) {
        await notifyAssignedHuman(ctx, row)
        continue
      }

// DESPUÉS
      if (row.assigned_to) {
        await notifyAssignedHuman(ctx, row)
        processed++
        continue
      }
```
