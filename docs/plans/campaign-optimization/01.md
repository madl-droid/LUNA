# Plan 1: UTM Foundation — Detection, Auto-creation, Schema

## Objetivo
Implementar el backend completo de deteccion UTM: schema, parseo, auto-creacion de campanas, deteccion dual (UTM > keyword), extension webhook, recording con match_source, toggles admin, y promptContext injection.

## Principio central
**UTM SIEMPRE GANA.** Si hay UTM y keyword, UTM manda. Keyword es backup para cuando no hay UTM.

## Tarea 3: UTM Parser utility

**Archivo NUEVO**: `src/modules/marketing-data/utm-parser.ts`

Funcion pura que extrae UTM params de texto que contenga URLs.

```typescript
/**
 * Extrae parametros UTM de URLs encontradas en texto.
 * Retorna el primer set de UTMs encontrado (prioridad: primer URL con utm_campaign).
 * Si no hay URLs con UTMs, retorna null.
 */
export function extractUtmFromText(text: string): UtmParams | null
```

### Logica:
1. Regex para encontrar URLs en el texto: `https?://[^\s<>"']+`
2. Para cada URL encontrada, parsear con `new URL(url)`
3. Extraer params que empiecen con `utm_` del `searchParams`
4. Si alguna URL tiene `utm_campaign` → retornar esos UTMs (primera URL gana)
5. Si ninguna URL tiene `utm_campaign` pero alguna tiene otros `utm_*` → retornar esos
6. Si no hay UTMs en ninguna URL → retornar `null`
7. Envolver en try/catch por si la URL es malformada

**Funcion auxiliar exportada**:
```typescript
/**
 * Parsea un objeto plano de UTMs (para webhook body).
 * Normaliza keys a lowercase, filtra vacios.
 */
export function normalizeUtmData(raw: Record<string, string>): UtmParams | null
```

**NO importar dependencias externas.** Usar `URL` nativo de Node.js.

### 5b. Nuevo metodo `matchUtm`

```typescript
/**
 * Intenta match por UTM: busca campana por utm_campaign, auto-crea si no existe.
 * Retorna CampaignMatchResult con matchSource y utmData, o null si no hay UTMs.
 */
async matchUtm(
  utmData: UtmParams,
  channelName: string,
  channelType: string,
): Promise<CampaignMatchResult | null>
```

Logica:
1. Si `channelType === 'voice'` → return null (mantener gate existente)
2. Si no hay `utmData.utm_campaign` → return null (sin utm_campaign no podemos mapear)
3. Buscar campana: `this.campaignQueries.findByUtmCampaign(utmData.utm_campaign)`
4. Si encontrada → retornar match con `matchSource: 'url_utm'`, score: 1.0
5. Si no encontrada → `this.campaignQueries.autoCreateFromUtm(utmData.utm_campaign, utmData)` → retornar match con `matchSource: 'url_utm'`, score: 1.0
6. Despues de auto-crear → recargar indice del matcher (por si le ponen keyword despues)

### 5c. Actualizar `match()` existente

### 5d. NO mezclar logica de UTM y keyword en el mismo metodo
`match()` = keyword puro (sincronico, como esta hoy).
`matchUtm()` = UTM puro (async, necesita DB).
La orquestacion de prioridad (UTM > keyword) se hace en el caller (intake.ts).

### 7b. Nueva logica de deteccion (reemplaza el cuerpo actual)

```typescript
async function detectCampaign(...): Promise<CampaignInfo | null> {
  const channelSvc = registry.getOptional<{get(): {channelType: string}}>(`channel-config:${channelName}`)
  const channelType = channelSvc?.get()?.channelType ?? 'instant'
  const roundNumber = sessionMessageCount + 1

  // --- Paso 1: Intentar UTM match (prioridad maxima) ---
  const { extractUtmFromText } = await import('../../modules/marketing-data/utm-parser.js')
  const utmFromText = extractUtmFromText(normalizedText)

  if (utmFromText) {
    type UtmMatchFn = (utmData: Record<string, string>, ch: string, ct: string) => Promise<CampaignMatchResult | null>
    const utmMatchFn = registry.getOptional<UtmMatchFn>('marketing-data:match-campaign-utm')
    if (utmMatchFn) {
      const utmResult = await utmMatchFn(utmFromText, channelName, channelType)
      if (utmResult) {
        return {
          id: utmResult.campaignId,
          visibleId: utmResult.visibleId,
          name: utmResult.name,
          keyword: utmResult.keyword,
          utm: utmResult.utmData,
          promptContext: utmResult.promptContext,
          matchScore: utmResult.score,
          matchSource: utmResult.matchSource,
        }
      }
    }
  }

  // --- Paso 2: Fallback a keyword match ---
  type KeywordMatchFn = (text: string, ch: string, ct: string, round: number) => CampaignMatchResult | null
  const keywordMatchFn = registry.getOptional<KeywordMatchFn>('marketing-data:match-campaign')
  if (!keywordMatchFn) return null

  const result = keywordMatchFn(normalizedText, channelName, channelType, roundNumber)
  if (!result) return null

  return {
    id: result.campaignId,
    visibleId: result.visibleId,
    name: result.name,
    keyword: result.keyword,
    utm: null,
    promptContext: result.promptContext,
    matchScore: result.score,
    matchSource: 'keyword',
  }
}
```

### 7c. Actualizar llamada a detectCampaign (linea 209)
Cambiar de:
```typescript
const detectedCampaign = detectCampaign(registry, normalizedText, message.channelName, session.messageCount)
```
A:
```typescript
const detectedCampaign = await detectCampaign(registry, normalizedText, message.channelName, session.messageCount)
```

**NOTA**: `detectCampaign` se llama ANTES del `Promise.allSettled` de linea 222. Es una sola llamada await, no paralela con el context loading. Esto es correcto porque:
- UTM match puede crear campanas (necesita DB write) → no conviene paralelizar con reads
- Es rapido (<10ms para busqueda, <20ms si auto-crea)
- El paralelismo de context loading (historial, memoria, knowledge) no se ve afectado

## Riesgos
1. **Import dinamico**: `import('../../modules/marketing-data/utm-parser.js')` en intake.ts. Si falla, usar import estatico al inicio del archivo.
2. **Race condition en auto-creacion**: Dos webhooks simultaneos con el mismo utm_campaign podrian intentar crear la misma campana. Mitigacion: `ON CONFLICT (utm_keys) DO NOTHING` o simplemente usar el ON CONFLICT del INSERT para que el segundo falle silenciosamente y haga lookup.
3. **Backward compatibility**: Los metodos `recordMatch` y servicios de match agregan parametros opcionales con defaults, por lo que callers existentes no se rompen.
