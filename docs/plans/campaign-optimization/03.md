## Tarea 2: ON CONFLICT en autoCreateFromUtm [CRITICO]

**Archivo**: `src/modules/marketing-data/campaign-queries.ts`

### 2a. Agregar unique index parcial en `ensureTables()`

Despues del GIN index existente (linea ~57), agregar:
```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_campaigns_auto_utm_name
  ON campaigns (name) WHERE origin = 'auto_utm'
```

Esto evita duplicados en campanas auto-creadas (mismo nombre = mismo utm_campaign). Las campanas manuales no se ven afectadas (origin='manual' no tiene constraint).

### 2b. Cambiar INSERT en `autoCreateFromUtm` (linea ~302)

De:
```sql
INSERT INTO campaigns (name, keyword, utm_keys, utm_data, origin, active, updated_at)
VALUES ($1, NULL, ARRAY[$1], $2, 'auto_utm', true, now())
RETURNING id, name, visible_id
```
A:
```sql
INSERT INTO campaigns (name, keyword, utm_keys, utm_data, origin, active, updated_at)
VALUES ($1, NULL, ARRAY[$1], $2, 'auto_utm', true, now())
ON CONFLICT (name) WHERE origin = 'auto_utm'
  DO UPDATE SET utm_data = EXCLUDED.utm_data, updated_at = now()
RETURNING id, name, visible_id
```

Asi: si ya existe una campana auto_utm con ese nombre, actualiza los UTM data y retorna la existente.

### 2c. Actualizar CLAUDE.md del modulo

Linea que dice "Auto-creacion usa ON CONFLICT para evitar race conditions" — ahora sera verdad.

## Tarea 3: Try-catch en dynamic import de intake.ts [CRITICO]

**Archivo**: `src/engine/boundaries/intake.ts`

### Opcion elegida: try-catch con graceful fallback

Envolver el bloque UTM completo (lineas ~930-953) en try-catch. Si el import falla o la funcion falla, retornar null (continuar al keyword match).

El import dinamico es el patron correcto aqui — otros archivos del engine ya lo usan (ej: linea 217 de context-builder.ts importa `lead-scoring/scoring-engine.js`). La diferencia es que los otros tienen try-catch. Este no lo tiene.

**Cambio en `detectCampaign()`:**

```typescript
async function detectCampaign(...): Promise<CampaignInfo | null> {
  const channelSvc = registry.getOptional<{get(): {channelType: string}}>(`channel-config:${channelName}`)
  const channelType = channelSvc?.get()?.channelType ?? 'instant'
  const roundNumber = sessionMessageCount + 1

  // --- Paso 1: Intentar UTM match (prioridad maxima) ---
  try {
    const { extractUtmFromText } = await import('../../modules/marketing-data/utm-parser.js')
    const utmFromText = extractUtmFromText(normalizedText)

    if (utmFromText) {
      // ... (bloque UTM existente sin cambios) ...
    }
  } catch {
    // marketing-data module not available or UTM parsing failed — continue to keyword
  }

  // --- Paso 2: Fallback a keyword match ---
  // ... (sin cambios) ...
}
```

**NO mover a servicio del registry** (hallazgo 7). Razon: el patron de import dinamico ya se usa en el engine (scoring-engine, etc.). Moverlo a servicio agrega complejidad sin beneficio real — el modulo marketing-data ya se comunica via servicios para match/recording. El UTM parser es una utilidad pura sin estado que no necesita lifecycle del registry. El try-catch resuelve el riesgo real.

## Sobre hallazgo 7 (import directo = violacion de politica)

**Decision: NO implementar.** El patron de import dinamico con try-catch es usado en otros lugares del engine (context-builder.ts importa scoring-engine.js en linea 217). El UTM parser es una utilidad pura sin estado — exponerla como servicio del registry agrega complejidad sin beneficio funcional. El riesgo real (crash si modulo no existe) se resuelve con el try-catch de Tarea 3. Si en el futuro se establece una politica mas estricta, se migrara con los demas imports dinamicos del engine.

## Riesgos
- **Unique index parcial**: `ON campaigns (name) WHERE origin = 'auto_utm'` requiere PostgreSQL 9.5+. El proyecto usa PG 14+, asi que no es un problema.
- **LOWER en busqueda**: Performance mínima ya que el GIN index de utm_keys sigue siendo el path principal. El LOWER solo aplica al valor entrante ($1), no a la columna.
