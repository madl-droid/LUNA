## Tarea 2: Content truncation en docs-read (D2)

**Archivo:** `tools.ts` — tool `docs-read` handler (mismo que Tarea 1)

### Cambio

Agregar truncation al body DESPUÉS de calcular wordCount pero ANTES de retornar:

```typescript
handler: async (input) => {
  const doc = await docs.getDocument(input.documentId as string)

  // Métricas sobre el documento completo
  const wordCount = doc.body.split(/\s+/).filter(Boolean).length
  const charCount = doc.body.length

  // Truncar si excede 30K chars
  const MAX_DOC_CHARS = 30_000
  let body = doc.body
  let truncated = false
  if (body.length > MAX_DOC_CHARS) {
    body = body.slice(0, MAX_DOC_CHARS)
      + `\n\n[... documento truncado: mostrando ${MAX_DOC_CHARS.toLocaleString()} de ${charCount.toLocaleString()} caracteres (${wordCount.toLocaleString()} palabras totales)]`
    truncated = true
  }

  return {
    success: true,
    data: {
      ...doc,
      body,
      wordCount,
      charCount,
      truncated,
    },
  }
},
```

### Por qué 30K chars y no 8K

El agentic loop trunca tool results a 8K chars (`src/engine/agentic/agentic-loop.ts:442`). Pero esa truncación es ciega — corta el JSON a 8K y no deja indicador útil.

Al truncar a 30K **antes** de serializar:
1. El indicador `[... documento truncado]` sobrevive (está al final del body)
2. El body truncado + metadata JSON cabe en ~32K chars serializado
3. El agentic loop luego trunca a 8K, pero el agente ya sabe que el doc original tenía N chars/palabras
4. Si se aumenta el límite del agentic loop en el futuro, el doc ya viene pre-truncado a un tamaño razonable

### Notas
- `30_000` chars ≈ 7,500 tokens — un doc de 30K chars es suficiente para la mayoría de documentos de trabajo
- El word count refleja el documento COMPLETO, no el truncado

## Tarea 3: Batch Edit para Docs (D1)

### Paso 3a: Tipo nuevo

**Archivo:** `types.ts` — después de `DocInfo` (línea ~122)

```typescript
export interface DocEditOperation {
  type: 'append' | 'insert' | 'replace'
  text: string                // texto a insertar/agregar, o replaceText para replace
  searchText?: string         // solo para type='replace': texto a buscar
  index?: number              // solo para type='insert': posición (1-based, como Google Docs API)
}
```

### Paso 3b: Método en servicio

**Archivo:** `docs-service.ts` — nuevo método en clase `DocsService`

```typescript
async batchEdit(
  documentId: string,
  operations: DocEditOperation[],
): Promise<{ applied: number }>
```

#### Lógica de implementación

**IMPORTANTE — Orden de operaciones en Google Docs batchUpdate:**
- Las requests se ejecutan secuencialmente (cada una ve el resultado de la anterior)
- `replaceAllText` es global (busca por texto, no por índice) — se puede poner en cualquier orden
- `insertText` es sensible al índice — inserciones en índices menores desplazan el contenido posterior

**Estrategia segura:**

```
1. Separar operaciones en dos grupos:
   a. REPLACES: ops con type='replace' → generan replaceAllText requests
   b. INSERTS/APPENDS: ops con type='insert' o 'append' → generan insertText requests

2. Para los REPLACES:
   - Construir: { replaceAllText: { containsText: { text: op.searchText, matchCase: true }, replaceText: op.text } }
   - Orden no importa

3. Para los APPENDS:
   - Pre-fetch del endIndex del documento: documents.get → body.content → último element.endIndex
   - Para CADA append consecutivo, acumular offset:
     firstAppend:  insertText at index = endIndex - 1
     secondAppend: insertText at index = endIndex - 1 + lengthOfFirstText
     etc.
   - Esto es necesario porque batchUpdate ejecuta secuencialmente y cada insertText
     desplaza el endIndex

4. Para los INSERTS (con índice específico):
   - Ordenar de MAYOR a MENOR índice
   - Insertar en ese orden (así las inserciones de mayor índice no afectan las de menor índice)
   - Construir: { insertText: { location: { index: op.index }, text: op.text } }

5. Orden final de requests en el batchUpdate:
   a. Primero todos los replaceAllText (no afectan índices entre sí)
   b. Luego inserts ordenados de mayor a menor índice
   c. Finalmente appends con offset acumulado

   NOTA: Los replaces cambian la longitud del documento, lo que afecta los
   índices de insert. Para evitar problemas, si hay MEZCLA de replaces e inserts,
   hacer en dos batchUpdate calls separados:
   - Call 1: todos los replaces
   - Call 2: re-fetch endIndex, ejecutar inserts + appends

   Si solo hay replaces O solo hay inserts/appends, una sola call basta.

6. Ejecutar documents.batchUpdate con las requests
7. Retornar { applied: operations.length }
```

## Tarea 4: Retry con backoff en OAuth init (A1)

**Archivo:** `manifest.ts` — función init() (líneas 443-451)

### Estado actual
```typescript
if (config.GOOGLE_CLIENT_ID && config.GOOGLE_CLIENT_SECRET) {
  try {
    await oauthManager.initialize()
  } catch (err) {
    logger.warn({ err }, 'OAuth initialization failed — connect manually from console')
  }
}
```

### Cambio

Reemplazar con retry loop de 3 intentos con exponential backoff:

```typescript
if (config.GOOGLE_CLIENT_ID && config.GOOGLE_CLIENT_SECRET) {
  let initialized = false
  for (let attempt = 1; attempt <= 3 && !initialized; attempt++) {
    try {
      await oauthManager.initialize()
      initialized = true
    } catch (err) {
      if (attempt < 3) {
        const delayMs = 2000 * Math.pow(2, attempt - 1) // 2s, 4s, 8s
        logger.warn({ err, attempt, nextRetryMs: delayMs }, 'OAuth initialization failed — retrying')
        await new Promise(resolve => setTimeout(resolve, delayMs))
      } else {
        logger.warn({ err, attempts: 3 }, 'OAuth initialization failed after 3 attempts — connect manually from console')
      }
    }
  }
}
```

### Notas
- 3 intentos: inmediato, +2s, +4s. Total máximo de espera: ~6s. No bloquea significativamente el bootup.
- Si los 3 fallan, el comportamiento es idéntico al actual: OAuth queda desconectado, tools no se registran, el usuario puede conectar manualmente desde la console.
- El backoff ayuda con: network transitorio al arrancar, token refresh temporalmente fallido, Google API rate limit en cold start.
- NO cambiar la lógica post-init (registro de tools, etc.) — solo afectar el `oauthManager.initialize()` call.
