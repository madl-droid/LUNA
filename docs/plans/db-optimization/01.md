# Plan 1: Canonical Schema Blueprint

**Tipo:** Investigacion + documentacion
**Dependencias:** Ninguna (ejecutar PRIMERO)
**Output:** `docs/plans/db-optimization/schema-blueprint.md`

---

## Objetivo

Producir EL documento de referencia definitivo con el schema completo de la base de datos para beta. Cada tabla, cada columna, cada tipo, cada constraint, cada indice, cada FK. Este documento es la "constitucion" que los Plans 2, 3, 4 y 5 implementan.

**IMPORTANTE:** Este plan NO escribe codigo. Solo produce documentacion.

---

## Contexto

El schema actual esta disperso en 2 fuentes:
1. **51 archivos en `src/migrations/*.sql`** — tablas fundacionales con ALTERs incrementales
2. **~19 archivos TypeScript** en modulos — tablas de modulo creadas en `init()`

El blueprint debe reconciliar ambas fuentes y producir el estado FINAL deseado, no el estado actual.

### Decisiones ya tomadas (NO cambiar):
- `companies` → MANTENER (B2B futuro)
- `agents` → DROP
- `task_checkpoints` → DROP
- `session_summaries` v1 → DROP (reemplazada por v2)
- `conversation_archives` v1 → DROP (reemplazada por session_archives)
- `summary_chunks` → DROP (absorbida en session_memory_chunks)
- `llm_daily_stats` → MANTENER
- TEXT/VARCHAR → UUID donde aplique (ver overview.md)
- Agregar FKs faltantes (ver overview.md)
- `session_memory_chunks` absorbe `summary_chunks` via `source_type` discriminador
- `knowledge_chunks` permanece separada
- Todas las tablas (migracion + modulo) van en un solo squash

Crear `docs/plans/db-optimization/schema-blueprint.md` con este formato EXACTO para cada tabla:

```markdown
### {numero}. {nombre_tabla}
**Owner:** migration | {nombre-modulo}
**Lifecycle:** permanent | retention-based | session-scoped

| Column | Type | Nullable | Default | Constraint |
|--------|------|----------|---------|------------|
| id | UUID | NO | gen_random_uuid() | PRIMARY KEY |
| ... | ... | ... | ... | ... |

**Indexes:**
- `idx_name` ON table(col) [WHERE condition]

**FK Constraints:**
- column → target_table(target_col) [ON DELETE action]

**CHECK Constraints:**
- column IN ('val1', 'val2', ...)

**Seeds:** (si aplica)
- descripcion de datos iniciales

**Notes:** cualquier nota sobre cambios vs estado actual
```
