# Plan 4: Module Init Cleanup

**Tipo:** Refactor de codigo (TypeScript)
**Dependencias:** Plan 2 (v1→v2 code done), Plan 3 (migration creates all tables)

---

## Objetivo

Eliminar TODOS los `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, `ALTER TABLE ADD COLUMN`, y cualquier DDL SQL de los modulos. La unica fuente de verdad para el schema es `src/migrations/001_beta-schema.sql`.

Despues de este plan:
- Ningun modulo ejecuta DDL (CREATE/ALTER/DROP) en su init()
- Los archivos pg-store.ts, manifest.ts, db.ts solo tienen funciones de query (INSERT/SELECT/UPDATE/DELETE)
- La aplicacion arranca limpiamente porque la migracion ya creo todo

## Pasos de ejecucion

### Paso 1: Para CADA archivo, seguir este proceso

1. **Leer el archivo completo**
2. **Identificar TODOS los bloques de DDL** (CREATE TABLE, CREATE INDEX, ALTER TABLE, DO blocks con DDL)
3. **Verificar que el DDL corresponde a tablas en la migracion** — si un modulo crea una tabla que NO esta en 001_beta-schema.sql, es un bug (agregar a la migracion via Plan 3, o reportar discrepancia)
4. **Eliminar el DDL** sin romper la logica de las funciones de query
5. **Eliminar funciones wrapper** como `ensureTable()`, `ensureSchema()`, `runMigrations()` y sus llamadas
6. **Actualizar init()** del manifest si llamaba a ensureTable/ensureSchema
7. **Verificar que los imports no se rompan** al eliminar funciones

### Paso 2: Verificar llamadas a funciones eliminadas

Buscar en todo el codebase referencias a funciones que se eliminaron:
- `ensureTable`
- `ensureSchema`
- `ensureTables`
- `runMigrations`
- `createTables`

Ninguna debe existir despues de la limpieza.

### Paso 3: Verificar SQL queries que usen nombres/tipos actualizados

Si Plan 2 cambio nombres de tabla (ej: conversation_archives → session_archives), verificar que los queries de los modulos usen los nombres correctos. En particular:
- `src/modules/memory/pg-store.ts` — debe haber sido actualizado en Plan 2
- Cualquier modulo que haga queries JOIN con tablas de memoria

### Paso 4: Actualizar tipos TypeScript si columnas cambiaron

Si algun tipo de columna cambio (TEXT → UUID), los tipos TypeScript que mapean esas columnas pueden necesitar actualizacion. Verificar:
- `src/modules/memory/types.ts` — session_id fields
- `src/modules/hitl/types.ts` — contact_id, session_id fields (si existen)
- Otros types.ts que referencien columnas corregidas

### Paso 5: Verificar que la aplicacion puede arrancar

El flujo de arranque es:
1. `kernel/db.ts` → create pool (ya NO crea tablas)
2. `kernel/migrator.ts` → ejecuta 001_beta-schema.sql (crea TODO)
3. `kernel/setup/handler.ts` → setup wizard (ya NO crea tablas)
4. Module loader → cada modulo.init() (ya NO crean tablas)
5. Modulos hacen queries → tablas ya existen

Verificar que ningun modulo DEPENDA de que su init() cree tablas antes de que otro modulo haga queries.

### Paso 6: Actualizar CLAUDE.md de modulos afectados

Para modulos que tenian documentacion sobre "tablas creadas en init()", actualizar:
- `src/modules/memory/CLAUDE.md` — ya cubierto en Plan 2
- `src/modules/knowledge/CLAUDE.md` — si referencia ensureSchema
- Otros CLAUDE.md que mencionen "tablas creadas por el modulo"

Actualizar a: "Tablas definidas en 001_beta-schema.sql, queries en pg-store.ts"

### Paso 7: Compilar TypeScript

```bash
docker run --rm -v /docker/luna-repo:/app -w /app node:22-alpine npx tsc --noEmit
```

Corregir TODOS los errores antes de commit.

## REGLAS para el ejecutor

1. **NO modificar `src/migrations/001_beta-schema.sql`** — si falta una tabla, reportar como discrepancia
2. **NO eliminar funciones de query** (INSERT/SELECT/UPDATE/DELETE) — solo DDL
3. **Ser conservador con los seeds** — si un modulo seedea datos en init() (no en DDL), dejarlo. Solo eliminar si el seed ya esta en la migracion
4. **Verificar cada eliminacion** — leer el contexto completo antes de borrar un bloque
5. **Si un modulo tiene logica condicional** ("si la tabla no existe, crearla, luego migrar datos"), toda esa logica se elimina — la tabla ya existe via migracion
