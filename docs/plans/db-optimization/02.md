# Plan 2: v1→v2 Memory Consolidation

**Tipo:** Refactor de codigo (TypeScript)
**Dependencias:** Ninguna (toda la info necesaria esta en este documento)
**Paralelo con:** Plan 3 (no comparten archivos — Plan 2 es TypeScript, Plan 3 es SQL)

---

## Objetivo

Eliminar el dual-write v1/v2 del sistema de memoria. Despues de este plan:
- **NINGUN codigo** referencia `session_summaries` (v1), `conversation_archives` (v1), ni `summary_chunks`
- La compresion escribe SOLO a `session_archives`, `session_summaries_v2`, y `session_memory_chunks`
- El hybrid search lee SOLO de `session_summaries_v2` y `session_memory_chunks`
- El nightly batch opera SOLO sobre tablas v2

---

## Contexto critico

### Flujo ACTUAL de compresion (dual-write):
```
1. archiveSession()     → INSERT conversation_archives (v1)
2. saveSessionSummary() → INSERT session_summaries (v1)
3. saveChunks()         → INSERT summary_chunks (v1)
4. [compression-worker] → INSERT session_archives (v2)
5. [compression-worker] → INSERT session_summaries_v2 (v2)
6. [compression-worker] → INSERT session_memory_chunks (v2)
```

### Flujo TARGET (v2-only):
```
1. archiveSession()     → INSERT session_archives (v2)
2. saveSummary()        → INSERT session_summaries_v2 (v2)
3. saveChunks()         → INSERT session_memory_chunks (source_type='session_summary')
```

### Archivos que cambian:
- `src/modules/memory/pg-store.ts` — queries SQL (principal)
- `src/modules/memory/memory-manager.ts` — orquestacion de compresion + hybrid search

### Paso 4: Consolidar chunks (summary_chunks → session_memory_chunks)

**Archivo:** `src/modules/memory/pg-store.ts`

**ELIMINAR metodos de summary_chunks:**
- `saveChunks()` — reescribir para insertar en `session_memory_chunks` con `source_type = 'session_summary'`
- `getChunksBySummary()` — reescribir contra `session_memory_chunks` filtrando por `source_id`

- `searchChunksFTS()` — reescribir contra `session_memory_chunks` (tiene columna `tsv`)

**Mapping de columnas (summary_chunks → session_memory_chunks):**

| summary_chunks | session_memory_chunks | Notas |
|----------------|----------------------|-------|
| summary_id | source_id | UUID del summary como TEXT |
| contact_id | contact_id | Igual |
| chunk_text | content | Rename |
| chunk_index | chunk_index | Igual |
| embedding | embedding | Igual (vector 1536) |
| — | source_type | NUEVO: 'session_summary' |
| — | content_type | NUEVO: 'text' |
| — | session_id | NUEVO: obtener del summary |
| — | chunk_total | NUEVO: total de chunks del summary |
| — | embedding_status | NUEVO: 'pending' → 'embedded' |

**IMPORTANTE:** Al insertar chunks de session summary en `session_memory_chunks`:
- `source_type = 'session_summary'`
- `content_type = 'text'`
- `source_id = summaryId` (el UUID del session_summaries_v2 como string)
- `session_id = sessionId` (el UUID de la sesion)
- Los queries de busqueda deben filtrar por `source_type` cuando sea necesario

**CUIDADO con mezcla de datos:**
- Chunks de `source_type = 'session_summary'` son resumen de conversacion
- Chunks de `source_type = 'attachment'` o `source_type = 'message'` son contenido multimodal
- Los queries de hybrid search del memory-manager deben funcionar con ambos tipos (esto es correcto — queremos buscar en toda la memoria)
- Pero `getChunksBySummary()` debe filtrar por `source_id = summaryId` para no traer chunks de otros sources
