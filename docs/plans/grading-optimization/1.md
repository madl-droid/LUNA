# Plan 1: Core — Simplificación del Config y Scoring

## Objetivo
Eliminar multi-framework, simplificar la configuración a un framework activo por tenant, y arreglar problemas fundamentales del scoring engine.

## Pre-requisitos
- Ninguno. Este plan es la base para los demás.

## Contexto para el ejecutor

El módulo `src/modules/lead-scoring/` tiene un sistema multi-framework que soporta 3 frameworks simultáneos (CHAMP/B2B, SPIN/B2C, CHAMP+Gov/B2G) con detección de tipo de cliente por LLM. Esta complejidad no se justifica: el 99% de tenants usa un solo tipo de venta. La detección de client_type requiere una llamada LLM extra que es puro desperdicio.

**Principio guía**: un tenant elige UN preset al configurar, puede personalizarlo (agregar/quitar criterios), y ese es su framework. Si cambia de giro, cambia de preset. No hay routing automático.

## Cambios detallados

### 1. `src/modules/lead-scoring/types.ts`

**Eliminar:**
- `FrameworkType` type union (`'champ' | 'spin' | 'champ_gov'`)
- `ClientType` type union (`'b2b' | 'b2c' | 'b2g'`)
- `CLIENT_TYPE_FRAMEWORK` mapping
- `FrameworkPreset` interface (se fusiona con la config)
- `FrameworkConfig` interface (se fusiona con QualifyingConfig)

**Modificar `QualifyingCriterion`:**
```typescript
export interface QualifyingCriterion {
  key: string
  name: { es: string; en: string }
  type: 'text' | 'boolean' | 'enum'
  options?: string[]
  priority: 'high' | 'medium' | 'low'      // NUEVO — reemplaza weight
  required: boolean
  neverAskDirectly: boolean
  stage?: string
  enumScoring?: 'indexed' | 'presence'      // NUEVO — default 'indexed'
}
```

Nota: eliminar el campo `weight: number` de QualifyingCriterion. El peso se calcula en runtime desde `priority`.

**Simplificar `QualifyingConfig`:**
```typescript
export interface QualifyingConfig {

  preset: string | null                       // 'champ' | 'spin' | 'champ_gov' | null
```

**Eliminar:** `StatusMetric`, `MetricChannelBreakdown` (mover a pg-queries.ts como tipos locales si se necesitan).

**Mantener sin cambios:** `FrameworkStage`, `DisqualifyReason`, `QualifyingThresholds`, `QualificationStatus`, `ScoreResult`, `CriterionScore`, `StageScoreSummary`, `ExtractionResult`, `LeadSummary`, `LeadDetail`, `LeadScoringConfig`, `FrameworkObjective`, `generateKeyFromName()`.

### 2. `src/modules/lead-scoring/frameworks.ts`

Cambiar el concepto de "presets que se instancian como FrameworkConfig" a "presets que generan QualifyingConfig parcial".

**Nuevo tipo de preset:**
```typescript
export interface PresetDefinition {
  key: string                                 // 'champ', 'spin', 'champ_gov'
  name: { es: string; en: string }
  description: { es: string; en: string }
  defaultObjective: FrameworkObjective
  stages: FrameworkStage[]
  criteria: QualifyingCriterion[]             // usa priority en vez de weight
  disqualifyReasons: DisqualifyReason[]
  essentialQuestions: string[]
}
```

**Migrar cada preset:**
- Convertir `weight` a `priority`:
  - weight >= 8 → `'high'`
  - weight >= 4 → `'medium'`
  - weight < 4 → `'low'`
- Agregar `enumScoring: 'presence'` a criterios donde el orden no es semántico:
  - `purchase_model` (CHAMP): `'presence'`
  - `prior_experience` (SPIN): `'presence'`
  - `lead_source` (SPIN): `'presence'`
  - `entity_type` (CHAMP+Gov): `'presence'`
  - `procurement_type` (CHAMP+Gov): `'presence'`
  - `funding_source` (CHAMP+Gov): `'presence'`
  - Los demás enums que son escalas (low/medium/high, urgency, etc.) mantienen `'indexed'` (default)

**Reducir criterios por preset a max 10:**
- CHAMP: mantener los 10 más relevantes de los 16 actuales. Eliminar: `decision_maker_name` (raro que se diga), `purchase_model` (bajo valor), `scope_quantity` (muy específico), `expected_next_step` (el agente ya lo maneja), `competitors_evaluated` (bajo peso), `industry` (bajo peso). Quedan 10.
- SPIN: mantener los 10 más relevantes de los 16 actuales. Eliminar: `problem_duration`, `previous_attempts`, `lead_source`, `preferred_schedule`, `preferred_contact_method`, `current_solution`. Quedan 10.
- CHAMP+Gov: mantener los 10 más relevantes de los 24 actuales. Elegir los más críticos de cada stage. Eliminar todos los de weight <= 3 y los redundantes. Quedan 10.

**Exportar:**
```typescript
export const PRESETS: Record<string, PresetDefinition> = {
  champ: CHAMP_PRESET,
  spin: SPIN_PRESET,
  champ_gov: CHAMP_GOV_PRESET,
}

export function getPreset(key: string): PresetDefinition | undefined {
  return PRESETS[key]
}
```

### 3. `src/modules/lead-scoring/config-store.ts`

**Simplificar drásticamente:**

- Eliminar `setFramework()`, `resetFrameworkToPreset()`, `getActiveFrameworks()`, `getFrameworkForClientType()`, `isMultiFramework()`
- Eliminar `migrateOldConfig()` — reemplazar con nueva migración

**Nueva migración que maneja 3 formatos de entrada:**

```typescript
private migrateIfNeeded(raw: Record<string, unknown>): QualifyingConfig {
  // Format 1: Old flat (BANT) — has 'criteria' at root, no 'frameworks', no 'preset'
  if ('criteria' in raw && !('frameworks' in raw) && !('preset' in raw)) {
    return this.migrateFromFlat(raw)
  }
  // Format 2: Multi-framework (v2) — has 'frameworks' array
  if ('frameworks' in raw) {
    return this.migrateFromMultiFramework(raw)
  }
  // Format 3: New single-framework (v3) — has 'preset' key
  return raw as unknown as QualifyingConfig
}
```

- `migrateFromFlat()`: toma criteria/thresholds/disqualifyReasons del formato viejo, asigna preset='spin' (default), convierte weight→priority, genera stages desde SPIN preset.
- `migrateFromMultiFramework()`: toma el primer framework activo (o el primero si ninguno activo), extrae sus campos a nivel raíz.
- Ambas migraciones guardan el archivo migrado a disco.

**Nuevo `DEFAULT_CONFIG`:**
```typescript
const DEFAULT_CONFIG: QualifyingConfig = {
  preset: 'spin',
  objective: 'schedule',
  stages: SPIN_PRESET.stages,
  criteria: SPIN_PRESET.criteria,
  disqualifyReasons: SPIN_PRESET.disqualifyReasons,
  essentialQuestions: SPIN_PRESET.essentialQuestions,
  thresholds: { cold: 30, qualifying: 31, qualified: 70 },
  minConfidence: 0.4,
  dataFreshnessWindowDays: 90,
}
```

**Nuevos métodos:**
```typescript
applyPreset(presetKey: string): QualifyingConfig  // reemplaza criterios/stages/disqualify con el preset
addCriterion(criterion: QualifyingCriterion): void  // max 10 total
removeCriterion(key: string): void
```

**Validación simplificada:**
- No validar que weights sumen 100 (ya no existen)
- Validar que criterios ≤ 10
- Validar que enum criteria tengan options
- Validar que stages referenciados existan
- Validar essentialQuestions (max 2, referencian criterios válidos)

### 4. `src/modules/lead-scoring/scoring-engine.ts`

**Eliminar:**
- `resolveFramework()` — ya no hay multi-framework
- Toda referencia a `FrameworkConfig`, `ClientType`, `CLIENT_TYPE_FRAMEWORK`

**Modificar `calculateScore()`:**

Firma nueva:
```typescript
export function calculateScore(
  qualificationData: Record<string, unknown>,
  config: QualifyingConfig,
): ScoreResult
```

Cambios internos:
- Leer criterios directamente de `config.criteria` (no de framework)
- Calcular pesos desde priority:
  ```typescript
  const PRIORITY_WEIGHT: Record<string, number> = { high: 3, medium: 2, low: 1 }
  const totalWeight = criteria.reduce((sum, c) => sum + PRIORITY_WEIGHT[c.priority ?? 'medium'], 0)
  // Para cada criterio: normalizedWeight = (PRIORITY_WEIGHT[c.priority] / totalWeight) * 100
  ```

**Modificar `calculateCriterionPoints()`:**
- Para type `'enum'`: verificar `criterion.enumScoring`
  - `'presence'` (o undefined con tipo no-escala): dar `maxPoints` si está lleno
  - `'indexed'` (default): mantener lógica actual `(idx + 1) / options.length * maxPoints`

**Modificar `getCurrentStage()`:**
- Firma: `getCurrentStage(qualData, config: QualifyingConfig)` — lee stages y criteria de config directamente

**Modificar `buildQualificationSummary()`:**
- Quitar resolveFramework() — leer directo de config
- Quitar detección de client_type / multi-framework messaging
- Agregar objetivo al summary
- Simplificar: ya no necesita sección de "client type detection needed"

**Modificar `mergeQualificationData()`:**
- Agregar tracking de timestamps: al mergear un campo nuevo, guardar `_extracted_at[key] = new Date().toISOString()`
- Los campos `_client_type` y su manejo desaparecen (ya no hay multi-framework)
- Mantener `_disqualified` y `_confidence`

**Mantener sin cambios:** `resolveTransition()`, `VALID_TRANSITIONS`, `isFilled()`
