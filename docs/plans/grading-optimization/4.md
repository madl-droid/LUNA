### BUG-1: CRITICAL — XSS en `openDetail()` via contactId sin escapar
**Archivo:** `src/modules/lead-scoring/ui/lead-scoring.html` ~línea 1141

**Problema:** `contactId` se inyecta raw en atributos `onclick`:
```javascript
return `<tr onclick="openDetail('${l.contactId}')">`
```
Si el contactId contiene `')`, ejecuta JS arbitrario.

**Fix:** Reemplazar inline onclick con `data-id` attribute + event delegation. En el `renderLeads()`:

```javascript
return `<tr data-contact-id="${esc(l.contactId)}" class="lead-row">
  ...
  <td><button class="btn btn-ghost btn-small" data-contact-id="${esc(l.contactId)}">${t('detail')}</button></td>
</tr>`
```

Agregar event delegation al final del script (una sola vez, en init o DOMContentLoaded):
```javascript
document.getElementById('leads-tbody').addEventListener('click', function(e) {
  const row = e.target.closest('[data-contact-id]')
  if (row) {
    e.stopPropagation()
    openDetail(row.dataset.contactId)
  }
})
```

Eliminar TODOS los inline `onclick="openDetail('...')"` y `onclick="event.stopPropagation();openDetail('...')"` del renderLeads().

**Verificar también:** que `esc()` está definida en el HTML. Ya debería existir (~línea 1285). Confirmar que escapa `"`, `'`, `<`, `>`, `&`.

### BUG-2: HIGH — Confidence tracking con condición invertida
**Archivo:** `src/modules/lead-scoring/scoring-engine.ts` ~línea 327-334

**Problema:** La condición `merged[key] === extracted[key]` intenta verificar si el valor fue adoptado, pero:
1. Si el valor existente era igual al extraído, reporta "adoptado" aunque no se procesó
2. Para objetos/arrays, `===` compara referencia, siempre false

**Fix:** Trackear keys adoptadas con un Set en el loop principal:

```typescript
export function mergeQualificationData(
  existing: Record<string, unknown>,
  extracted: Record<string, unknown>,
  confidence: Record<string, number>,
  minConfidence = 0.4,
): Record<string, unknown> {
  const merged = { ...existing }
  const now = new Date().toISOString()
  const adoptedKeys = new Set<string>()     // ← NUEVO: trackear keys adoptadas

  for (const [key, newValue] of Object.entries(extracted)) {
    if (key.startsWith('_')) {
      merged[key] = newValue
      continue
    }

    const existingValue = existing[key]
    const newConfidence = confidence[key] ?? 0.5
    const existingConfidence = (existing['_confidence'] as Record<string, number> | undefined)?.[key] ?? 0
    const hasExisting = isFilled(existingValue)

    if (newConfidence < minConfidence && hasExisting) {
      continue
    }

    if (!hasExisting || newConfidence > existingConfidence) {
      merged[key] = newValue
      adoptedKeys.add(key)                   // ← NUEVO: marcar como adoptada

      // Track extraction timestamp
      const extractedAt = (merged['_extracted_at'] as Record<string, string>) ?? {}
      extractedAt[key] = now
      merged['_extracted_at'] = extractedAt
    }
  }

  // Update confidence tracking — SOLO para keys que realmente se adoptaron
  const existingConf = (existing['_confidence'] as Record<string, number>) ?? {}
  const acceptedConf: Record<string, number> = {}
  for (const [key, conf] of Object.entries(confidence)) {
    if (key.startsWith('_') || adoptedKeys.has(key)) {   // ← FIX: usar Set
      acceptedConf[key] = conf
    }
  }
  merged['_confidence'] = { ...existingConf, ...acceptedConf }

  return merged
}
```

**Cambio clave:** `merged[key] === extracted[key]` → `adoptedKeys.has(key)`

### BUG-3: HIGH — Tool description no se actualiza en hot-reload

### BUG-4: MEDIUM — Enum scoring da 50% a valores no reconocidos
**Archivo:** `src/modules/lead-scoring/scoring-engine.ts` ~líneas 238-243

**Problema:** Si el LLM extrae un valor que no está en las opciones del enum (typo, sinónimo), el lead recibe 50% de los puntos. Eso infla el score artificialmente.

**Fix:** Cambiar `maxPoints * 0.5` a `0` en ambos modos:

```typescript
case 'enum': {
  if (!criterion.options || criterion.options.length === 0) return 0
  const strVal = String(value)
  const idx = criterion.options.indexOf(strVal)

  // Presence mode: full points if valid option, 0 if unknown
  if (criterion.enumScoring === 'presence') {
    return idx !== -1 ? maxPoints : 0     // ← FIX: era maxPoints * 0.5
  }

  // Indexed mode (default): higher index = better score
  if (idx === -1) return 0                 // ← FIX: era maxPoints * 0.5
  const ratio = (idx + 1) / criterion.options.length
  return maxPoints * ratio
}
```

### BUG-5: MEDIUM — Nightly batch OFFSET pagination race condition
**Archivo:** `src/engine/proactive/jobs/nightly-batch.ts` ~línea 133-143

**Problema:** `ORDER BY ac.updated_at DESC` con OFFSET — si un lead se actualiza durante la iteración (score recalculado → updated_at = NOW()), el ORDER cambia y leads se saltan o procesan doble.

**Fix:** Cambiar a cursor-based pagination por `contact_id` (estable, inmutable):

```typescript
let lastContactId: string | null = null

while (true) {
  const params: unknown[] = [batchSize]
  let whereExtra = ''
  if (lastContactId) {
    whereExtra = ` AND ac.contact_id > $2`
    params.push(lastContactId)
  }

  const result = await ctx.db.query(
    `SELECT ac.contact_id,
            COALESCE(ac.qualification_data, '{}') AS qualification_data,
            ac.qualification_score
     FROM agent_contacts ac
     JOIN contacts c ON c.id = ac.contact_id
     WHERE c.contact_type = 'lead'
       AND ac.lead_status = 'cold'${whereExtra}
     ORDER BY ac.contact_id ASC
     LIMIT $1`,
    params,
  )

  if (result.rows.length === 0) break

  // ... procesamiento igual ...

  lastContactId = result.rows[result.rows.length - 1]!.contact_id
  total += result.rows.length
  if (total >= 1000) break
}
```

Eliminar la variable `offset` completamente.

### BUG-6: MEDIUM — `applyPreset()` no actualiza el objective
**Archivo:** `src/modules/lead-scoring/config-store.ts` ~línea 63-72

**Problema:** El spread `...this.config` preserva el `objective` anterior. Si cambias de SPIN (objective: sell) a CHAMP, mantiene `sell` en vez de usar el default del preset.

**Fix:** Agregar `objective: preset.defaultObjective` al config nuevo:

```typescript
applyPreset(presetKey: string): QualifyingConfig {
  const preset = PRESETS[presetKey]
  if (!preset) throw new Error(`Unknown preset: ${presetKey}`)
  const newConfig: QualifyingConfig = {
    ...this.config,
    preset: presetKey,
    objective: preset.defaultObjective,      // ← FIX: aplicar default del preset
    stages: JSON.parse(JSON.stringify(preset.stages)) as typeof preset.stages,
    criteria: JSON.parse(JSON.stringify(preset.criteria)) as typeof preset.criteria,
    disqualifyReasons: JSON.parse(JSON.stringify(preset.disqualifyReasons)) as typeof preset.disqualifyReasons,
    essentialQuestions: [...preset.essentialQuestions],
  }
  this.save(newConfig)
  return this.config
}
```

**Nota:** Si el admin quiere un objective diferente al default del preset, lo cambia después. Pero al aplicar un preset nuevo, el default del preset es lo correcto.
